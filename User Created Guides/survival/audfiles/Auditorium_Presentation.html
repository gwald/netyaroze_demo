Pradip Fatehpuria
Pradip Fatehpuria (2): Please always keep the PlayStation resources in mind when you are applying all the above in the NetYaroze universe. Try to keep the game simple but playable. For example, try to keep the 3D models in your game simple with minimal number of polygons - you have plenty of memory (1 MB VRAM) available for the textures and display/draw buffers, but the models remain in the main memory. Try putting all your music notes and sound effects in less than 512K, which is pretty much enough to generate good quality music.

   Slide 16: OS/Runtime Library Thu Aug 28 19:28:19 PDT 1997 
  Software Resources

Presented By
Pradip Fatehpuria
Pradip Fatehpuria: The crux of the PlayStation is it's OS and the runtime library. As I said previously, the PlayStation architecture provides a complex environment but most of the complexity is made transparent to you by means of a powerful and very efficient OS and the run-time library. The OS resides in the system ROM and the runtime library is loaded into first 0x90000 bytes of main memory when you boot the black PlayStation with the boot disc provided. The run-time library remains memory resident and your programs call various functions in this library to perform most of the tasks.

   Slide 38: Object Handling Thu Aug 28 19:44:05 PDT 1997 
  Object Handling

Presented By
Pradip Fatehpuria
Pradip Fatehpuria: Next I will talk about how I handled various types of objects in the world. There are five types of objects in the world. - The walls of the cubical in the world - The stationary wooden slabs - The enemy objects - The enemy remains when they dye (small balls come out of the enemy when they die) - And finally the flying robot which the player controls. All the objects are handled by means of a common object data structure called OBJECT and is defined in LibObjc.h. Common object handling routines are stored in LibObjc.c.
Greg Labrec leaves auditorium.
Question: Whats a good rule as to how large your packet area should be?
Answer: A simple a good rule is to precalculate number of each type of primitive in each model in the world and use the size taken by them as the size of the packet area. The number of primitives in the world can be found when you use RSD2TMD tool, which tells you the number of each primitive type in the final TMD.

   Slide 43: Collision Detection Thu Aug 28 19:50:00 PDT 1997 
  Collision Detection

Presented By
Pradip Fatehpuria
Pradip Fatehpuria: Each object is assumed to be enclosed inside a bounding box. The size of the bounding box is pre-determined (stored in the table for the 3D-models) and is saved in their OBJECT structure when we initialize them. This bounding box is later used to determine the collision between objects.
 


   Slide 44: Collision Detection 2 Thu Aug 28 19:50:34 PDT 1997 
  Collision Detection

Presented By
Pradip Fatehpuria
Pradip Fatehpuria: The main game loop handles the object movement. A separate handler for each object is called from the main game loop. This handler determines the displacement to be applied to the object for the next frame. Before the displacement is applied to the object, we check if the object's path will cross with any other object in the world. Each object type has it's own function to determine its collision with other objects in the world, which is called by the object handler.
 


   Slide 45: Collision Detection 3 Thu Aug 28 19:51:09 PDT 1997 
  Collision Detection

Presented By
Pradip Fatehpuria
Pradip Fatehpuria: In case there is a collision, a flag is returned which tells it how the collision happened in terms of the direction of the object and the truncated displacement vector. The appropriate action is taken according to the flag returned and the new displacement vector is returned which is then applied to the object to generate the new object position. This process is performed to all objects in the world.
